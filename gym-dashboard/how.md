# Building a Live Gym Dashboard from a Google Sheet

I track every workout in a Google Sheet. One row per set: date, day type (Push/Pull/Leg), exercise, set number, weight, reps, volume, and notes. After a while of this, the sheet had grown into something genuinely interesting, there were patterns in there I couldn't see by scrolling.

So I built a dashboard. Not a notebook, not a one-off chart in Excel, a live page on my portfolio that pulls the latest data every day and renders it with no third-party chart libraries. Here's how it works, why I made the choices I did, and what I'd do differently.

## The constraint that shaped everything

I wanted the dashboard hosted as part of my static portfolio site. No backend server, no database, no paid service. That single constraint drove every other decision: the data needs to land somewhere a static page can fetch it from, and the page itself needs to do all the work in the browser.

The cleanest answer turned out to be: keep the source of truth in Google Sheets (where I already log workouts on my phone), and use GitHub Actions as a free cron job that snapshots the sheet to a CSV in a public repo every night. The frontend then fetches that CSV directly from `raw.githubusercontent.com`. No server, no auth on the read path, no rate limits I'd ever realistically hit.

## The pipeline

There are really three pieces: a Python script that pulls the sheet, a GitHub Action that runs the script on a schedule, and a frontend that reads the resulting CSV.

### 1. Pulling from Google Sheets

The Python script is short, maybe 20 lines. It uses `gspread` with a service account, opens the sheet by ID, and dumps all values to `output.csv`:

```python
creds_json = os.environ["GOOGLE_CREDENTIALS_JSON"]
creds_info = json.loads(creds_json)
creds = Credentials.from_service_account_info(creds_info, scopes=scopes)
client = gspread.authorize(creds)

spreadsheet = client.open_by_key(os.environ["GOOGLE_SHEET_ID"])
data = spreadsheet.sheet1.get_all_values()

with open("output.csv", "w", newline="", encoding="utf-8") as f:
    csv.writer(f).writerows(data)
```

The two secrets, the service account JSON and the sheet ID, live in GitHub repo secrets. I shared the sheet with the service account's email address as a viewer. That's the entire auth setup.

### 2. The cron job

GitHub Actions has a `schedule` trigger that takes a cron expression. I run it at midnight Eastern (5:00 UTC):

```yaml
on:
  schedule:
    - cron: "0 5 * * *"
  workflow_dispatch:
```

The workflow checks out the repo, installs Python deps, runs the pull script, then commits the updated CSV back to the repo if anything changed:

```yaml
- name: Commit and push output.csv
  run: |
    git config user.name "github-actions[bot]"
    git config user.email "github-actions[bot]@users.noreply.github.com"
    git add output.csv
    git diff --cached --quiet || git commit -m "Update output.csv [automated]"
    git push
```

The `git diff --cached --quiet || git commit` pattern is the trick that keeps the commit history clean, no empty "nothing changed today" commits on rest weeks.

### 3. The frontend

This is where most of the actual code lives. The dashboard fetches the CSV at page load:

```javascript
const CSV_URL = 'https://raw.githubusercontent.com/Streetlight321/pull_gym_data/refs/heads/main/output.csv';
const resp = await fetch(CSV_URL, { cache: 'no-store' });
const text = await resp.text();
const raw = parseCSV(text);
```

I wrote a tiny RFC-4180-compliant CSV parser by hand because some of my notes have commas in them and I didn't want to pull in a library. The parser walks the line character by character, tracking whether it's inside quotes:

```javascript
if (inQuotes) {
  if (ch === '"' && line[i + 1] === '"') { field += '"'; i += 2; }
  else if (ch === '"') { inQuotes = false; i++; }
  else { field += ch; i++; }
}
```

After parsing, the data goes through a cleaning pass that handles all the quirks of a sheet that's been edited by a human for a couple years: dates in `M/D/YYYY` format, `"Body Weight"` strings in the weight column, `#VALUE!` errors leaking out of broken formulas, `"Legs"` getting normalized to `"Leg"`, that kind of thing.

## Why no chart library?

This was the part I went back and forth on. I have nothing against Chart.js or Recharts, I've used both, but for this project I wanted something I could fully reason about. Every line, every dot, every tooltip is hand-rolled SVG, generated by code like:

```javascript
const c = svg('circle', {
  cx: p.x, cy: p.y, r: 4,
  fill: color,
  class: 'dot'
});
attachTip(c, `${g} · ${formatDate(p.date)}: ${fmt(Math.round(p.val))}`);
```

The payoff: zero dependencies, the page loads instantly, and when I want to do something weird, like animate the line drawing itself in by stroking along a calculated path length, I can just do it. The CSS handles the actual animation:

```css
.chart-container svg polyline {
  stroke-dasharray: var(--dash, 2000);
  stroke-dashoffset: var(--dash, 2000);
  animation: draw 1.1s var(--ease-out) forwards;
}
```

The JS measures each polyline's actual path length and sets `--dash` per-chart, so short lines and long lines both draw in at the same rate. Small detail, but it's the kind of thing you can't easily get out of a charting library without fighting it.

## What the dashboard actually shows

There are four panels:

The **hero stats** at the top show all-time totals, sessions, total volume, most-trained group, heaviest lift. These never change with the date filter; they're always the lifetime view.

**Daily Volume by Type** is the main chart. Three colored lines (Push purple, Pull pink, Leg teal) showing volume per session. There's a 30d / 90d / All-time toggle that re-renders the chart on click. Click any dot and it expands a card below with every set from that workout.

**Exercise Progress** lets you pick any exercise from a filtered dropdown and see weight (or reps, for bodyweight movements) over time. It also computes a 3-session trend, up, down, or flat, and flags PRs.

**Volume by Muscle Group** is a simple animated horizontal bar chart, totals across the visible range.

**Recent Sessions** is a clickable log of the last 7 workouts. Each one expands into a full set-by-set table.

## What I'd do differently

A few things I'd change if I were starting over.

The CSV fetch is unauthenticated and uncached, which is fine for one user but would fall apart immediately if the page got real traffic. Putting a CDN like Cloudflare in front of the raw GitHub URL would solve that for free.

The cleaning logic is scattered across the JS file. `parseDate`, `parseWeight`, `parseReps`, `normalizeGroup`, these should really live in the Python step, so the CSV the frontend gets is already clean. Right now the frontend is doing schema work that should have happened upstream.

And I'd probably add a derived "PRs over time" view. The data's all there; I just haven't built the panel yet.

## The bigger lesson

The whole stack, Sheets → Actions → CSV → static page, costs nothing, requires no servers, and updates itself every night. The most expensive part of building it was deciding I didn't need a backend. Once that was settled, every other piece fell into place.

If you've got data sitting in a spreadsheet that you'd like to see as a live dashboard, this pattern works. Service account, cron'd Action, raw CSV fetch, vanilla JS. Three moving parts. That's it.
